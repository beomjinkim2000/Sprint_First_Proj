## 마일스톤 3 실험 전략

> 실험 순서: Q3(진단) → Q2 결과 확인 → Q1 → Q4 → Q5 → Q6/Q7 (병렬 가능)

### Q1. 앙상블: 약한 모델 제거하면 더 오르나?
**가설** V2.7.3(val 0.9506), V4-2(val 0.9616) 포함 시 WBF 박스 위치를 흐려서 하락  
**비교 A** WBF-v1: 9모델 전체 (val 0.95~0.97 포함), iou=0.5, skip=0.25 → **Kaggle 0.95646**  
**비교 B** WBF-top: val mAP 0.97↑ 모델만 (V2.7.5, V2.7.6, V4-3, YOLOv8x), 동일 iou/skip/weights  
**확인** B > A면 모델 선별 효과 / 아니면 WBF 파라미터 문제

### Q2. WBF 파라미터가 최적인가?
**가설** iou=0.5는 인접 알약 박스를 잘못 병합할 수 있고, skip=0.25는 노이즈 포함  
**비교 A** WBF-v1: iou=0.5, skip=0.25 → **Kaggle 0.95646**  
**비교 B** WBF-v2: iou=0.55, skip=0.3, 동일 9모델/weights → 결과 대기 중

### Q3. YOLO11x val(좀 더 높음) vs test(좀 더 낮음) 괴리가 과적합인가? ✅ 진단 도구 구현 완료
**가설** train loss는 내려가는데 val loss가 올라가면 과적합, 둘 다 내려가면 distribution shift  
**진단 방법** YOLOv8x ep150 재학습 시 `metrics_v3-x.csv`의 train vs val box/cls/dfl loss 곡선 확인  
**결과에 따른 분기**
- val loss도 함께 수렴 → distribution shift → Q7(데이터 추가) 우선
- val loss가 train 대비 발산 → 과적합 → Q5(epoch 연장) 재고, Q6(1024) 신중

### Q4. YOLO11x가 YOLOv8x보다 강한가?
**근거** Q5(epoch 연장) 전에 어떤 아키텍처를 키울지 결정해야 compute 낭비 없음  
**비교 A** YOLOv8x ep350: wd=0.0001, phase3 head lr=0.00001, backbone lr=0.000001, batch=8, img_size=640 → **val 0.9679(EMA), Kaggle 0.93430**  
**비교 B** YOLO11x ep350: 동일 하이퍼파라미터 (wd=0.0001, head lr=0.00001, backbone lr=0.000001, batch=8, img_size=640)  
**선행** Q3 결과로 과적합 패턴 파악 후 필요 시 wd 조정 반영  
**추가 분석** 두 모델의 클래스별 F1 비교 → 어떤 아키텍처가 어떤 유형 알약에서 강한지 파악 → [[concepts/F1 Score]]

### Q5. epoch 연장이 성능을 올리나?
**근거** Q4 우승 아키텍처 확정 후 해당 아키텍처로 연장 — YOLO11x가 이기면 YOLOv8x 500ep는 낭비  
**비교 A** Q4 우승 아키텍처 ep350 결과 (Q4에서 확정)  
**비교 B** 동일 아키텍처, 동일 하이퍼파라미터, ep350 체크포인트에서 이어받아 ep500까지 추가 학습  
**선행** Q3(과적합 진단) + Q4(아키텍처 선정) 완료 후 진행

### Q6. 해상도 1024가 성능 개선되나?
**근거** 알약 각인·텍스트 세부 특징이 640에서 손실될 수 있음  
**비교 A** Q4 우승 아키텍처, ep350, img_size=640, 동일 하이퍼파라미터 → Q4 결과를 기준으로 사용  
**비교 B** 동일 아키텍처, 동일 하이퍼파라미터, img_size=1024, batch_size=4 (메모리 때문에 절반)  
**선행** Q4 아키텍처 확정 후 진행  
**추가 분석** Q4에서 클래스별 F1이 낮은 클래스들이 1024에서 개선되는지 집중 확인 → 각인·텍스트로 구분되는 클래스가 대상 → [[concepts/F1 Score]]

### Q7. AI Hub 추가 데이터가 성능을 높이나?
**근거** Q3 val loss가 val/train 모두 낮은데 test가 낮으면 → 데이터 분포 문제 → 다양한 데이터 필요  
**비교 A** Q4 우승 아키텍처, ep350, 기존 대회 데이터만 → Q4 결과를 기준으로 사용  
**비교 B** 동일 아키텍처, 동일 하이퍼파라미터, AI Hub 허용 데이터 추가 학습  
**조건** `TL_2_조합.zip`, `TS_2_조합.zip` 사용 금지 / 허용 데이터셋 목록 먼저 확인  
**선행** Q3에서 data-limited 여부 확인 후 진행 여부 결정  
**추가 분석** Q4 클래스별 F1이 낮은 클래스 = 데이터 부족 가능성 → 해당 클래스 AI Hub 데이터 우선 확보 대상 → [[concepts/F1 Score]]

### 제출 전 공통 단계: F1 곡선으로 최적 confidence threshold 확인
**적용 시점** 각 실험의 best 모델이 나올 때마다 1회  
**방법** conf=0.05~0.90 구간에서 val F1 계산 (IoU=0.5 고정) → F1 최대 지점 = 최적 conf  
**반영** `postprocess.conf_threshold` 업데이트 후 제출 / WBF의 `skip_box_thr`도 동일 방식 최적화 가능  
**근거** 현재 conf=0.25는 관행적 기본값, 모델마다 최적값이 다를 수 있음 → [[concepts/F1 Score]]
