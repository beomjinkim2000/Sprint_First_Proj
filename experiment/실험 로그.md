| Q4 | Q3 우승 아키텍처로 ep150 → 200 → 250 과적합 진단 | 계획 중 | Q3 | 과적합 예상 |
| Q5 | img_size 640 → 1024 | 계획 중 | Q4 | B(1024) > A |

### Q1. 앙상블 — 약한 모델 제거하면 더 오르나?

| | 설정 | 결과 |
|---|---|---|
| **A** | WBF-v1: 9모델 전체, iou=0.5, skip=0.25 | Kaggle **0.95646** |
| **B** | WBF-top: val 0.97↑ 4모델만, 동일 파라미터 | 계획 중 |

**가설** V2.7.3(0.9506) · V4-2(0.9616) 포함 시 WBF 박스 위치 합의를 흐려 하락  
**확인** B > A → 모델 선별 효과 / B ≤ A → WBF 파라미터 문제  
**예상** B > A. 낮은 모델이 틀린 박스에 가중치를 줘서 앙상블 평균을 흐림.

### Q2. WBF 파라미터가 최적인가?

|       | 설정                         | 결과                 |

| **A** | iou=0.55, skip=0.3         | Kaggle **0.95646** |
| **B** | iou=0.55, skip=0.3, 동일 9모델 | 대기 중               |

**가설** iou=0.5는 인접 알약 박스를 잘못 병합할 수 있고, skip=0.25는 노이즈 포함

### Q3. 아키텍처 비교 — YOLOv8x vs YOLO11x (ep150)

> 기존 ep350 결과 초기화. ep150으로 공정하게 재비교.

|       | 설정                                                                          | val mAP | Kaggle |
| ----- | --------------------------------------------------------------------------- | ------- | ------ |
| **A** | YOLOv8x, ep150, wd=0.0001, head lr=1e-5, backbone lr=1e-6, batch=8, img=640 | 계획 중    | 계획 중   |
| **B** | YOLO11x, ep150, 동일 하이퍼파라미터                                                  | 계획 중    | 계획 중   |

**판단** Kaggle test 점수로 우승 아키텍처 결정 → Q4로 넘김  
**추가 분석** 클래스별 F1 비교 — 어떤 아키텍처가 어떤 유형 알약에서 강한지 → [[concepts/F1 Score]]  
**예상** B(YOLO11x) > A. 최신 아키텍처라 특징 추출 효율이 높을 것.

### Q4. 과적합 진단 — Q3 우승 아키텍처로 ep150 / 200 / 250

> Q3 ep150 결과를 ep150 기준선으로 재활용. ep200, ep250은 이어받아 학습.

| | 설정 | val mAP | Kaggle |
|---|---|---|---|
| **ep150** | Q3 우승 아키텍처 ep150 (Q3 결과 재활용) | — | — |
| **ep200** | ep150 체크포인트 → ep200 이어받기 | — | — |
| **ep250** | ep200 체크포인트 → ep250 이어받기 | — | — |

**판단 기준**

| 패턴 | 해석 | 다음 행동 |
|---|---|---|
| test 점수가 ep마다 계속 오름 | 아직 plateau 전 | ep250 또는 그 이상 추가 탐색 |
| test 점수가 어느 ep에서 멈추거나 하락 | 과적합 시작 | 해당 ep = 최적 epoch 확정 |

**예상** 과적합. 높은 val mAP@50=1.0이 이미 이상 신호. test 점수는 ep200~250 구간에서 주춤할 것으로 예상.  
**확정값** Q4 최적 epoch → Q5·Q6 학습 epoch으로 사용

### Q5. 해상도 1024가 성능 개선되나?

| | 설정 | 결과 |
|---|---|---|
| **A** | Q3 우승 아키텍처, img=640, batch=8, ep=Q4 최적 | Q4 결과 그대로 |
| **B** | 동일 아키텍처, img=1024, batch=4, ep=Q4 최적 | 계획 중 |

**선행** Q4 최적 epoch 확정 후 진행  
**추가 분석** Q3에서 F1 낮은 클래스(각인·텍스트 유형)가 1024에서 개선되는지 집중 확인 → [[concepts/F1 Score]]  
**예상** B(1024) > A. 각인·텍스트 세부 특징이 살아남아 클래스 구분력 높아질 것. 단 batch=4 → 학습 안정성 주의.

### Q6. AI Hub 추가 데이터가 성능을 높이나?

| | 설정 | 결과 |
|---|---|---|
| **A** | Q3 우승 아키텍처, 기존 대회 데이터만, ep=Q4 최적 | Q4 결과 그대로 |
| **B** | 동일 아키텍처, AI Hub 허용 데이터 추가, ep=Q4 최적 | 계획 중 |

**조건** `TL_2_조합.zip` · `TS_2_조합.zip` 사용 금지 / 허용 데이터셋 목록 먼저 확인  
**선행** Q4 완료 후 진행  
**추가 분석** Q3 클래스별 F1이 낮은 클래스 = 데이터 부족 가능성 → 해당 클래스 AI Hub 데이터 우선 확보 → [[concepts/F1 Score]]  
**예상** B(추가 데이터) > A. 학습 데이터가 늘면 일반화 성능 향상. F1 낮은 클래스에 데이터 보강되면 test 점수 격차 줄어들 것.

### 공통: 제출 전 F1 곡선으로 최적 conf threshold 확인

- **시점** 각 실험의 best 모델이 나올 때마다 1회
- **방법** conf=0.05~0.90 구간에서 val F1 계산 (IoU=0.5 고정) → F1 최대 지점 = 최적 conf
- **반영** `postprocess.conf_threshold` 업데이트 후 제출 / WBF `skip_box_thr`도 동일 방식 최적화 가능
- **근거** conf=0.25는 관행적 기본값, 모델마다 최적값 다를 수 있음 → [[concepts/F1 Score]]
