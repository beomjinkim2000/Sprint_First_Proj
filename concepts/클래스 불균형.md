# 클래스 불균형 (Class Imbalance)

학습 데이터에서 클래스마다 샘플 수가 크게 다른 상태.  
많이 등장한 클래스 위주로 학습되고, 적게 등장한 클래스는 성능이 떨어지는 문제가 생긴다.

## 이 프로젝트 상황

- 56개 클래스, 학습 이미지 232장, annotation 763개
- 이상적 균등 분포: 클래스당 약 13.6회
- 가장 많은 클래스(일양하이트린정 2mg): 153회
- 가장 적은 클래스: 3회
- 상위 3개 클래스를 제외하면 비교적 균등하지만, **절대값 자체가 작아** 희귀 클래스는 특징 학습이 부족할 수 있다.

## 대응 방법 3가지

### 1. WeightedRandomSampler (현재 적용)

DataLoader가 이미지를 뽑을 때 희귀 클래스 이미지의 **뽑힐 확률**을 높이는 방식.  
Loss를 건드리지 않고 학습 데이터 구성 자체를 바꾼다.

```python
from torch.utils.data import WeightedRandomSampler

# 이미지별 가중치: 이미지 안의 가장 희귀한 클래스 기준
# class_weight[c] = max_count / count_c
# sample_weight[i] = max(class_weight[c] for c in 이미지 i의 클래스들)

sampler = WeightedRandomSampler(sample_weights, num_samples=len(sample_weights), replacement=True)
DataLoader(dataset, sampler=sampler, ...)  # shuffle=True 대신 sampler 사용
```

- 희귀 클래스 이미지가 에폭당 더 자주 뽑힌다
- PyTorch 내장 기능, 가장 깔끔한 방법

### 2. pos_weight (BCE에 직접 가중치)

PyTorch `BCEWithLogitsLoss`는 `pos_weight` 파라미터를 지원한다.  
희귀 클래스를 놓쳤을 때 손실을 더 크게 계산한다.

```python
# pos_weight[c] = max_count / count_c
# 희귀 클래스(3회): pos_weight = 153/3 = 51 → 손실 51배
nn.BCEWithLogitsLoss(pos_weight=torch.tensor([1.0, 51.0, ...]))
```

단, Ultralytics `v8DetectionLoss`는 BCE를 파라미터 없이 고정 생성하므로,  
생성 후 `criterion.bce`를 덮어써야 한다는 단점이 있다. (억지스러운 방식)

### 3. Focal Loss

틀린 클래스에 **자동으로** 가중치를 높이는 손실 함수. (RetinaNet 논문에서 제안)  
Ultralytics 코드 안에 `FocalLoss`가 이미 구현되어 있다.

## 방법 비교

| 방법 | 건드리는 것 | 깔끔함 | 비고 |
|---|---|---|---|
| WeightedRandomSampler | DataLoader (데이터 뽑는 방식) | ★★★ | PyTorch 내장 |
| pos_weight | 손실 함수 (BCE) | ★★ | Ultralytics에서 억지로 패치해야 함 |
| Focal Loss | 손실 함수 (전체 교체) | ★★ | 객체 탐지 표준 방식 |
